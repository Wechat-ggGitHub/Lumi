# Shrew Claude Code 页面改版执行规范

## 1. 文档目的

本文档用于指导 Claude Code 对 Shrew 的桌面端页面进行统一改版。目标不是自由发挥，而是让 Claude Code 严格按照本规范完成页面重构、视觉统一和交付。

Claude Code 在开始修改前，必须先完整阅读：

1. `/.monkeycode/docs/shrew-single-persona-prd.md`
2. 本文件 `/.monkeycode/docs/shrew-claude-code-ui-revision-spec.md`

如果实现与本文档冲突，以本文档为准；如果产品目标与本文档冲突，以 PRD 为准。

## 2. 执行目标

Claude Code 这次改版需要同时达成以下目标：

1. 把 Shrew 统一成一个围绕“单一 AI 分身”展开的桌面产品。
2. 保持主窗口首层就是聊天页，不改成会话列表，也不改成控制台首页。
3. 统一聊天页、分身设定、记忆管理、技能管理、服务连接、设置页的视觉语言。
4. 拆清设置体系，避免“模块导航页”和“具体配置页”混在同一页面。
5. 让技能管理和服务连接从空壳页面变成真实可配置、可管理的页面。
6. 输出结果必须达到“可继续迭代的中高保真静态页面”标准，而不是概念草图。

## 3. 严格产品约束

Claude Code 修改页面时，必须遵守以下产品边界：

1. 产品一级对象是“单一分身”，不是“聊天会话”。
2. 主窗口首层必须是持续聊天页。
3. 不允许新增显性的“新建对话”入口。
4. 用户通过 `/clear` 清空上下文。
5. 聊天前台只展示：
   - 用户消息
   - 分身消息
   - 轻量系统消息
6. 不在前台展示工具调用过程、推理过程、Agent 内部步骤。
7. Memory、技能、服务连接、设置都必须作为二级页面存在。
8. 分身设定页负责分身自身配置；设置页只负责系统级配置。
9. 技能管理页前台名称固定为“技能管理”。
10. MCP 前台名称固定为“服务连接”。

## 4. 严格视觉原则

Claude Code 不允许随意发挥成新的视觉方向，必须遵守以下视觉原则：

1. 全部页面统一为深色桌面端风格。
2. 视觉气质要克制、专业、现代，不要做成可爱聊天软件，也不要做成企业后台模板。
3. 页面重点是“分身关系感 + 执行感”，不是“功能堆叠感”。
4. 不允许出现左侧大导航栏布局；主窗口是单窗口内容结构。
5. 不允许把页面改成移动端比例。
6. 不允许使用过重的拟物、发光、玻璃炫技效果。
7. 不允许出现明显的“AI 生成后台页”既视感，例如：
   - 纯白大表单
   - 四处散落的标签页
   - 无主次的功能卡片瀑布
8. 所有二级页面必须共享统一 Header、内容区、底部操作区语言。

## 5. 页面尺寸与窗口规范

### 5.1 主窗口尺寸

Claude Code 必须以桌面端主窗口为目标，不要做响应式站点首页。

建议设计基准：

1. 窗口外框宽度：`920px`
2. 窗口外框高度：`640px`
3. 最小可接受宽度：`880px`
4. 最小可接受高度：`620px`
5. 窗口圆角：`20px`
6. 窗口阴影：中等，不夸张

### 5.2 内边距规范

1. 页面左右内边距：`24px`
2. 页面顶部内容区内边距：`20px`
3. 区块之间垂直间距：`20px`
4. 卡片内边距：`16px`
5. 小组件间距：`8px`
6. 大区块内元素间距：`12px`

### 5.3 可滚动区规范

1. 聊天消息区可滚动。
2. 记忆列表可滚动。
3. 技能列表可滚动。
4. 服务连接列表可滚动。
5. 设置详情页内容过长时，主体内容区滚动，Header 和底部主操作区尽量稳定。

## 6. 颜色规范

Claude Code 必须使用统一 Design Tokens，不要零散写颜色。

### 6.1 基础色板

```css
:root {
  --bg-app: #0b1020;
  --bg-window: #11172a;
  --bg-surface-1: #151d33;
  --bg-surface-2: #1a2440;
  --bg-surface-3: #202b49;
  --line-default: rgba(255, 255, 255, 0.08);
  --line-strong: rgba(255, 255, 255, 0.14);
  --text-primary: #eef2ff;
  --text-secondary: #c7d0ee;
  --text-muted: #97a3c7;
  --brand-primary: #7c9cff;
  --brand-primary-hover: #92adff;
  --brand-soft: rgba(124, 156, 255, 0.16);
  --success: #6ee7b7;
  --warning: #f6c177;
  --danger: #ff8a8a;
  --info: #7dd3fc;
}
```

### 6.2 使用规则

1. 主背景使用 `--bg-app`。
2. 主窗口内容背景使用 `--bg-window`。
3. 卡片背景优先使用 `--bg-surface-1` 或 `--bg-surface-2`。
4. 一级正文使用 `--text-primary`。
5. 次级说明使用 `--text-secondary`。
6. 弱提示与元信息使用 `--text-muted`。
7. 主按钮、选中态、焦点态使用 `--brand-primary`。
8. 不允许使用纯黑纯白作为大面积背景。
9. 危险操作只在真正删除、断开等动作中使用 `--danger`。
10. 不允许把多个颜色同时用作主强调色。

## 7. 字体与排版规范

### 7.1 字体栈

```css
font-family: Inter, ui-sans-serif, system-ui, -apple-system, BlinkMacSystemFont,
  "PingFang SC", "Hiragino Sans GB", "Microsoft YaHei", sans-serif;
```

### 7.2 字号层级

Claude Code 必须固定使用以下字号层级，不要任意扩散：

1. 窗口标题：`13px` / `500`
2. 页面标题：`28px` / `700`
3. 页面副说明：`14px` / `400`
4. 区块标题：`16px` / `600`
5. 卡片标题：`15px` / `600`
6. 正文：`14px` / `400`
7. 次级正文：`13px` / `400`
8. 标签与元信息：`12px` / `500`
9. 超小辅助文字：`11px` / `500`

### 7.3 行高规范

1. 页面标题：`1.2`
2. 正文：`1.6`
3. 次级说明：`1.5`
4. 列表项元信息：`1.4`

### 7.4 排版原则

1. 页面标题只能有一个。
2. 每页最多一个主描述文案。
3. 不要在一个页面里堆太多二级说明段落。
4. 长文本回答必须分段，并允许代码/路径高亮。

## 8. 圆角、边框、阴影规范

### 8.1 圆角

1. 窗口：`20px`
2. 大卡片：`16px`
3. 小卡片：`14px`
4. 按钮：`12px`
5. 输入框：`12px`
6. 标签 Chip：`999px`

### 8.2 边框

1. 所有输入框、卡片、按钮边界默认使用 `1px`。
2. 默认边框色使用 `--line-default`。
3. Hover 或 focus 可以增强到 `--line-strong`。
4. 不允许同时叠太多描边和发光。

### 8.3 阴影

1. 主窗口阴影中等即可。
2. 卡片阴影要尽量弱，以分层为主，不以漂浮感为主。
3. 主按钮可有非常轻的阴影，不允许夸张 glow。

## 9. 按钮规范

### 9.1 按钮类型

Claude Code 只允许使用三类按钮：

1. Primary Button
2. Secondary Button
3. Ghost Button

### 9.2 按钮尺寸

1. 默认高度：`36px`
2. 小按钮高度：`30px`
3. 图标按钮尺寸：`32px * 32px`
4. 左右内边距：`12px` 到 `14px`

### 9.3 按钮语义

1. 一个页面只能有一个最主要的 Primary Button。
2. “保存”“新增”“发送”这类强动作才可使用 Primary。
3. “返回”默认用 Ghost 或 Secondary，不可抢主操作。
4. 删除类操作默认不用高亮主按钮。

### 9.4 禁止项

1. 不允许一个页面出现多个同级 Primary Button。
2. 不允许把所有动作都做成胶囊大按钮。

## 10. 输入框与表单规范

### 10.1 输入框高度

1. 单行输入框：`40px`
2. 下拉选择框：`40px`
3. 多行输入框最小高度：`88px`

### 10.2 表单结构

每组表单必须遵循：

1. Label
2. Optional Helper Text
3. Control
4. Optional Feedback Text

### 10.3 表单规则

1. 已保存状态不要伪装成输入框默认值。
2. 凭证类字段必须显示状态，例如：`已保存`、`未配置`。
3. 错误提示必须贴近字段出现。
4. 不允许一页里混用“字段级保存”和“整页保存”。

## 11. 页面统一骨架规范

所有二级页面都必须遵循这个结构：

```text
Window Bar
Page Header
Main Content
Bottom Action Bar（如需要）
```

### 11.1 Window Bar

包含：

1. 左侧 traffic lights
2. 中间应用名 `Shrew`
3. 右侧不放复杂操作

### 11.2 Page Header

包含：

1. 返回按钮
2. 页面标题
3. 一句说明
4. 右上主操作或次操作

### 11.3 Main Content

1. 用卡片或区块组织内容。
2. 不允许出现无结构的大面积散点控件。

### 11.4 Bottom Action Bar

适用于设置、分身设定等编辑型页面：

1. 固定在底部或视觉稳定。
2. 只保留整页主操作，如 `保存更改`。

## 12. 聊天主页面改版规范

### 12.1 页面定位

聊天页是主窗口首层，是整个产品的主页面。

### 12.2 结构

必须包含：

1. 轻量 Header
2. 消息流
3. 底部输入区

### 12.3 Header 组成

1. 分身头像
2. 分身名称 `Shrew`
3. 一句人格描述或当前状态
4. 右上更多入口按钮

### 12.4 状态表达规则

1. 页面只保留一个主要状态表达位置。
2. 不允许顶部显示“正在思考”，底部又显示“处理中”这种重复状态。
3. 推荐状态放在 Header 副信息位置。

### 12.5 消息流规则

1. 用户消息右侧对齐。
2. 分身消息左侧对齐。
3. 系统消息居中弱化展示。
4. 长回答必须分段。
5. 路径、命令、代码片段必须有独立高亮样式。

### 12.6 输入区规则

1. 始终显示多行输入框。
2. 左侧可有 slash 提示，如 `/clear`。
3. 右侧保留语音入口和发送按钮。
4. `Enter` 发送、`Shift+Enter` 换行。
5. 如果 busy，也必须明确说明输入是否仍可用。

### 12.7 禁止项

1. 不允许添加会话列表。
2. 不允许添加推荐问题宫格。
3. 不允许把技能、服务连接、Memory 摘要塞到聊天首屏主区域。

## 13. 设置总页改版规范

### 13.1 页面定位

设置总页只做“系统设置目录页”。

### 13.2 结构

必须包含以下分组卡片：

1. 模型与凭证
2. 语音
3. 运行环境
4. 交互偏好
5. 数据与隐私

### 13.3 每张卡片必须包含

1. 分组名称
2. 当前摘要
3. 当前状态
4. 进入详情的箭头或按钮

### 13.4 禁止项

1. 不允许在设置总页顶部放“分身设定 / 记忆管理 / 技能管理 / 服务连接”的并列导航按钮。
2. 不允许在设置总页直接展开长表单。

## 14. 设置详情页改版规范

### 14.1 页面定位

设置详情页用于具体编辑某个系统级配置分组。

### 14.2 允许的详情页

至少应包含：

1. 模型与凭证详情页
2. 语音详情页
3. 运行环境详情页
4. 交互偏好详情页

### 14.3 表单规则

1. 整页只保留一个保存按钮。
2. 可以有“测试连接”“浏览目录”这类局部动作。
3. 但不能再出现多个“保存”按钮。

### 14.4 模型与凭证详情页必须包含

1. 服务商选择
2. 模型选择
3. API Key 状态
4. 输入新 Key 的输入框
5. 测试连接按钮
6. 错误或成功反馈区

### 14.5 语音详情页必须包含

1. 识别服务说明
2. Access Token 状态
3. 更新 Token 输入框
4. 语音静音超时 Slider
5. 当前秒数显示

### 14.6 运行环境详情页必须包含

1. 默认工作目录路径
2. 浏览按钮
3. 解释 Claude Code 将在此目录执行命令的说明

## 15. 分身设定页改版规范

### 15.1 页面定位

分身设定页用于配置分身身份和行为风格，不处理系统级参数。

### 15.2 页面结构

必须按以下顺序组织：

1. 基础身份
2. 人格表达
3. 协作偏好
4. 高级设置

### 15.3 基础身份区必须包含

1. 头像
2. 名称
3. 一句简介

### 15.4 人格表达区必须包含

1. 性格
2. 语气
3. 回答详略

### 15.5 协作偏好区必须包含

1. 澄清偏好
2. 工作方式

### 15.6 高级设置区必须包含

1. 可折叠的 `高级 System Prompt`

### 15.7 交互规则

1. 选项组必须清楚表达是单选。
2. 高级设置默认折叠。
3. 页面底部提供统一的 `保存更改`。

### 15.8 禁止项

1. 不允许把所有字段平铺成一张长表单而没有分组。
2. 不允许让高级 Prompt 抢主视觉。

## 16. 记忆管理页改版规范

### 16.1 页面定位

记忆管理页是“Shrew 记住了什么”的档案页，不是纯后台条目表。

### 16.2 页面结构

必须包含：

1. 顶部标题和 `新增记忆`
2. 记忆总览卡
3. 筛选行
4. 记忆列表

### 16.3 记忆总览卡必须包含

1. Shrew 当前记住了什么
2. 2 到 4 条总结性 bullet

### 16.4 记忆卡片必须包含

1. 类型标签，如 `习惯`、`事实`
2. 记忆正文
3. 来源
4. 时间
5. 状态
6. 文本操作：编辑、置顶、失效、删除

### 16.5 交互规则

1. 操作不要只用图标，至少关键操作要有文字。
2. `自动提炼` 必须体现为来源信息，而不是孤立灰字。
3. `删除` 与 `失效` 必须明确区分。

## 17. 技能管理页改版规范

### 17.1 页面定位

技能管理页必须表达“Shrew 当前具备哪些技能”，不是单纯的导入器。

### 17.2 页面结构

必须包含：

1. 已启用技能区
2. 待配置或异常区
3. 新增技能区

### 17.3 技能卡片必须包含

1. 技能名称
2. 技能说明
3. 当前状态
4. 最近使用或配置摘要
5. 操作按钮：配置、停用，必要时继续配置

### 17.4 新增技能区必须包含

1. 导入 `.md`
2. 导入 `.zip`
3. 导入文件夹
4. 辅助说明：需要包含 `SKILL.md`

### 17.5 交互规则

1. 首屏先看已启用技能，不要先看导入大框。
2. 技能详情可用侧边抽屉或展开卡片承载。
3. 必须能够承接真实 Claude Code skill 配置流。

### 17.6 禁止项

1. 不允许整页只剩一个虚线导入框。
2. 不允许把页面做成技能市场。

## 18. 服务连接页改版规范

### 18.1 页面定位

服务连接页必须表达“Shrew 能访问哪些外部服务”，不是底层技术空壳页。

### 18.2 页面结构

必须包含：

1. 说明卡
2. 已连接服务区
3. 空状态或列表态
4. 新增连接入口

### 18.3 服务卡必须包含

1. 服务名称
2. 用途说明
3. 连接状态
4. 最近检查时间或授权状态
5. 操作按钮：配置、测试连接、重连、断开

### 18.4 空状态必须回答

1. 服务连接是什么
2. 为什么当前为空
3. 下一步怎么开始

### 18.5 交互规则

1. 必须有 `测试连接`。
2. 前台弱化 `MCP` 术语，但技术流程不能省略。
3. 新增连接可以用抽屉或向导承载：
   - 选择类型
   - 填写配置
   - 授权/测试
   - 保存

## 19. 组件复用要求

Claude Code 必须尽量复用同一套组件，不要每页重做一遍。

必须抽出并统一以下组件：

1. Window Bar
2. Page Header
3. Section Header
4. Summary Card
5. List Card
6. Primary / Secondary / Ghost Button
7. Single Line Input
8. Textarea
9. Select
10. Chip Group
11. Status Badge
12. Empty State
13. Bottom Action Bar

## 20. 文案风格要求

1. 页面标题短而明确。
2. 页面副说明只写一句，不写长篇解释。
3. 配置字段说明尽量说人话，不要堆技术黑话。
4. 错误文案要明确下一步该做什么。
5. 不要把页面写成营销文案。

## 21. 代码实现要求

如果 Claude Code 输出的是 HTML/CSS/JS 或前端页面代码，必须遵守：

1. 所有颜色、间距、圆角、字号尽量使用 token 或 CSS 变量。
2. 页面结构语义清晰。
3. 不要为了做效果引入大量无关依赖。
4. 不要写过度复杂的动画。
5. Hover、focus、active 至少有基础反馈。
6. 所有按钮和输入框的状态要可区分。

## 22. 交付规范

Claude Code 最终交付至少需要满足以下标准：

1. 必须覆盖这 7 个页面：
   - 聊天主页面
   - 设置总页
