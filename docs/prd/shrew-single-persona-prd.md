# Shrew 单分身产品 PRD

## 1. 文档信息

- 产品名称：Shrew
- 文档主题：单分身主体验与桌面端产品化 PRD
- 文档版本：v0.1
- 文档状态：讨论稿
- 适用范围：本期单分身版本

## 2. 产品背景

Shrew 当前是一个基于 macOS 菜单栏的桌面应用，本质上是对 Claude Code 的桌面端产品化封装。用户可通过全局快捷键唤起语音输入，以自然语言向 Agent 发起请求，并通过菜单栏状态和主窗口查看结果。

本期产品需要从“语音驱动的 AI 编程助手快捷入口”进一步演进为“面向电脑端的单一 AI 助理分身”。产品不再以“对话会话”作为一级对象，而以“分身”作为一级对象，围绕分身的持续交流、长期记忆、能力管理和系统配置构建整体体验。

## 3. 产品目标

本期目标是让用户在菜单栏中打开 Shrew 时，感受到自己是在与一个持续存在、具有身份与记忆的 AI 助理互动，而不是在使用一个传统 chatbot 或会话管理工具。

核心目标包括：

1. 建立“单一分身”而非“多会话”的产品心智。
2. 让主窗口成为持续聊天主界面，而不是对话详情页。
3. 通过分身设定、Memory、技能、服务连接和设置，建立完整但克制的产品结构。
4. 保持对 Claude Code 能力的复用，不在本期过度重建底层 Agent 基础设施。

## 4. 本期范围

本期仅支持一个 AI 分身，不支持多分身创建、切换或并行管理。

该分身具备以下核心属性：

1. 名称
2. 头像
3. 简介
4. 性格与语气
5. 默认行为偏好
6. 默认技能集合
7. 长期记忆
8. 持续聊天历史

## 5. 本期非目标

为控制范围，本期明确不做以下能力：

1. 多分身创建、切换与管理
2. 面向用户的多会话列表模型
3. 显性“新建对话”主入口
4. 工具调用过程可视化
5. 面向普通用户的 MCP 技术细节配置台
6. Memory 候选审核流
7. 推荐问题、模板化起手式
8. 技能商店或技能市场
9. 语音发送后自动打开主窗口
10. 强头像化、强角色化的 chatbot 视觉风格

## 6. 核心产品原则

1. 产品主体是“单一 AI 分身”，不是“会话列表”。
2. 用户默认与该分身保持持续对话关系。
3. 主界面不提供强显性的“新建对话”入口。
4. 用户通过 `/clear` 清空当前上下文，而不是通过创建新会话开始新一轮交流。
5. 前台体验强调连续交流，后台实现允许上下文切段与任务归档，但默认不向用户显式暴露会话结构。
6. 首屏优先传达“这个分身是谁、现在在做什么、我刚刚和它聊到了哪里”。

## 7. 产品定位与用户心智

### 7.1 一级对象

Shrew 的一级对象是“分身”，不是“聊天会话”。

用户打开产品时，默认理解应为：

1. 我正在和一个持续存在的 AI 助理交流。
2. 这个助理有名字、头像、性格、技能和记忆。
3. 我不是在管理多个 session，而是在和同一个分身持续相处。

### 7.2 对话心智

本期采用单分身持续对话模式。

1. 聊天历史以前台连续时间流呈现。
2. 不提供显性新建对话入口。
3. 用户通过 `/clear` 清空上下文。
4. `/clear` 后仍处于同一条聊天流内。

### 7.3 前台连续、后台分段

用户前台看到的是一条持续聊天流。系统后台可基于以下时机进行上下文切段与任务归档：

1. 用户执行 `/clear`
2. 一次任务执行完成
3. 长时间无交互
4. 上下文过长需要重建
5. Memory 提炼或归档触发

这些内部结构主要服务于：

1. 上下文控制
2. 成本控制
3. Memory 提炼
4. 任务排查
5. 历史检索

但默认不作为前台主心智暴露给用户。

## 8. 用户故事

### 8.1 首次进入

1. 作为第一次使用 Shrew 的用户，我希望打开主窗口时看到的是一个正在等待我交流的分身，而不是复杂的配置面板，这样我能快速理解产品的核心体验。
2. 作为第一次使用 Shrew 的用户，我希望即使还没有历史聊天，也能直接在主窗口底部输入框开始交流，这样我不会被空页面阻塞。
3. 作为第一次使用 Shrew 的用户，我希望在需要时能轻量看到语音快捷入口和基础配置入口，但不希望被大量模板或推荐问题打扰。

### 8.2 语音快捷发起

1. 作为日常开发中的用户，我希望通过全局快捷键快速唤起底部语音条，这样我无需切换到主窗口也能发起请求。
2. 作为使用语音输入的用户，我希望在转写完成后还能编辑文字并按回车确认发送，这样我可以纠正识别错误后再提交。
3. 作为使用语音条的用户，我希望发送后主窗口不要自动弹出，这样不会打断我当前正在进行的工作。

### 8.3 主窗口持续聊天

1. 作为用户，我希望打开右上角入口后直接回到与分身的持续聊天页，这样我能立刻续聊，而不是先找某个会话。
2. 作为用户，我希望主聊天页底部始终有输入框，这样我在任何时候都可以继续打字交流。
3. 作为用户，我希望我发送的文字立即进入聊天流，这样我能确认消息已经发出。

### 8.4 上下文管理

1. 作为用户，我希望通过 `/clear` 清空当前上下文，而不需要创建一个新对话，这样更符合“我一直在和同一个分身交流”的感觉。
2. 作为用户，我希望 `/clear` 执行后能看到一条轻量系统消息确认已生效，这样我知道后续交流会从新的上下文开始。

### 8.5 Memory 使用

1. 作为用户，我希望 Shrew 能在任务结束后自动记住真正重要的长期信息，而不是把所有内容都机械存下来。
2. 作为用户，我希望能查看 Shrew 当前记住了什么，这样我能理解它为什么这样回应我。
3. 作为用户，我希望能手动修正错误记忆，这样长期关系不会被错误信息污染。

### 8.6 技能与服务连接

1. 作为用户，我希望看到 Shrew 当前有哪些技能，这样我知道它能做什么。
2. 作为用户，我希望看到 Shrew 当前连接了哪些服务，这样我知道它可以访问哪些外部能力来源。
3. 作为用户，我希望技能和服务连接是两个不同页面，这样我不会混淆“它会什么”和“它连到了什么”。

### 8.7 错误与状态反馈

1. 作为用户，我希望当请求失败时，由分身自然告诉我哪里出了问题，而不是只看到冰冷错误码。
2. 作为用户，我希望即使主窗口没开，也能通过菜单栏状态知道 Shrew 当前是否正在工作。

## 9. 整体信息架构

### 9.1 入口结构

Shrew 当前存在两个核心前台入口：

1. 底部悬浮语音条
2. 右上角主窗口

### 9.2 入口职责

1. 语音条负责快捷输入，包括录音、转写、编辑、发送确认。
2. 主窗口负责持续聊天、查看状态、查看历史、进入二级页。
3. 菜单栏负责全局状态反馈。

### 9.3 主窗口结构

主窗口采用“聊天为主，其他能力进入二级页”的结构。

首层页面聚焦：

1. 分身身份感知
2. 持续聊天
3. 当前状态反馈

二级页面承载：

1. 分身设定
2. Memory
3. 技能管理
4. 服务连接
5. 设置

## 10. 主窗口定义

点击右上角应用 icon 后，打开的应是“分身主窗口”，而不是传统意义上的“对话详情页”。

该主窗口的定位是：

1. 用户与单一分身持续交流的主界面
2. 查看分身身份、当前状态与最近聊天内容的核心入口
3. 进入分身设定、Memory、技能管理、服务连接与系统设置的统一入口

主窗口首层即聊天页，不再单独抽象一个先于聊天的“分身首页”。

## 11. 聊天主界面

聊天主界面是主窗口首层的核心内容区。

其目标是让用户感受到：

1. 自己正在与一个持续存在的 AI 分身交流
2. 交流历史是连续的，而不是被多个 chat session 打断
3. 分身既具有“人格感”，也具备“工作能力”

### 11.1 页面结构

聊天页由三部分组成：

1. 顶部轻量 Header
2. 中部连续聊天流
3. 底部常驻文字输入框

### 11.2 顶部 Header

聊天页顶部采用“轻量 Header + 可展开身份卡”的方式。

默认 Header 展示：

1. 分身头像
2. 分身名称
3. 当前状态点
4. 一行副信息

副信息采用动态切换策略：

1. 分身空闲时，展示一句人格描述
2. 分身忙碌时，展示当前状态描述

点击 Header 后，可展开完整身份卡，展示：

1. 完整简介
2. 性格摘要
3. 默认能力摘要
4. 进入分身设定页的入口

### 11.3 聊天历史呈现

聊天历史在前台以连续时间流方式呈现。

设计要求：

1. 历史记录按持续聊天流展示
2. 不在前台强调“会话”“线程”“上下文段”等结构
3. 用户应感知为始终在与同一个分身对话

### 11.4 消息角色

1. 聊天流采用轻角色感展示。
2. 用户消息与分身消息应有清晰但克制的视觉区分。
3. 系统消息采用更轻的样式，不应喧宾夺主。
4. 本期不做强头像墙、重标签、重动作化的 chatbot 视觉风格。

### 11.5 回复呈现

1. 分身回复在聊天流中采用流式生成。
2. 用户可以感知分身正在逐步组织和输出回复。
3. 流式过程中不展示工具调用、命令执行或其他内部执行细节。
4. 用户对执行过程的主要感知来自状态系统，而不是消息中的技术过程展开。

### 11.6 主聊天输入

1. 主聊天页底部常驻文字输入框。
2. 用户可在主聊天窗口中直接输入文字继续与分身交流。
3. 语音条与主聊天输入框共用同一条持续聊天流。
4. 主聊天输入框支持 slash 命令机制。
5. 本期先仅开放 `/clear`。
6. 主聊天输入框使用 `Enter` 发送消息。
7. 主聊天输入框使用 `Shift+Enter` 换行。

### 11.7 发送反馈

1. 用户在主聊天输入框发送文字后，用户消息立即进入聊天流。
2. 消息进入聊天流后，状态系统切换到 `正在思考` 或 `正在执行`。
3. 分身回复随后以流式形式逐步出现。

### 11.8 空状态

1. 空状态采用分身导向设计。
2. 页面应优先展示分身欢迎语与简短自我介绍。
3. 页面底部保留常驻文字输入框，允许用户直接开始聊天。
4. 可提供少量轻引导，如提示语音快捷键或基础配置入口，但不应让页面转向工具操作面板心智。
5. 空状态不展示推荐问题。
6. 空状态不展示模板化起手式。

## 12. 语音条体验

语音条是 Shrew 的快捷输入入口，不是主聊天窗口的替代品。其价值在于让用户能在不打开主窗口的情况下，以低打扰方式快速发起请求。

### 12.1 页面定位

语音条负责：

1. 录音
2. 转写
3. 文本编辑
4. 发送确认

主聊天窗口负责：

1. 查看持续聊天流
2. 查看状态
3. 继续键盘聊天
4. 进入各类二级页

### 12.2 核心流程

用户使用语音条的流程为：

1. 通过全局快捷键唤起语音条
2. 开始录音
3. 结束录音后进入转写
4. 转写结果展示在语音条中
5. 用户可编辑文本
6. 用户按回车确认发送
7. 请求提交给 Agent 执行
8. 主窗口若未打开，不自动弹出
9. 菜单栏状态用于反馈正在执行

### 12.3 交互原则

1. 语音条是快捷入口，不是唯一输入入口。
2. 用户确认发送前，不向主聊天流写入消息。
3. 语音条发送后，不自动打开主聊天窗口。
4. 用户如需查看详细聊天与历史，可主动打开右上角主窗口。

### 12.4 与主聊天页的关系

1. 语音条输入与主聊天页文字输入进入同一条持续聊天流。
2. 语音条负责“快速发起”。
3. 主聊天页负责“持续交流”。

## 13. `/clear` 与上下文管理

本期不提供显性的“新建对话”入口。

当用户希望开始一个新的上下文时，通过输入 `/clear` 完成。

`/clear` 的产品定义如下：

1. 清空当前上下文
2. 后续对话从新的上下文继续理解
3. 历史消息仍保留在连续时间流中
4. 系统内部开启新的上下文段，但前台不突出展示

`/clear` 执行后，聊天流中插入一条系统消息。

该系统消息的要求是：

1. 明确说明当前上下文已清空
2. 保留轻微分身表达感
3. 避免纯工具提示语气
4. 避免过度拟人化表达

推荐文案风格示例：

`当前上下文已清空，我会从新的话题继续理解你。`

## 14. 任务定义与任务完成

一次任务定义为：用户发出一次请求后，分身完成该次响应或执行，并给出最终答复。

该定义主要用于后台状态管理、Memory 提炼与任务归档，不改变前台持续聊天流的展示方式。

### 14.1 任务完成反馈

1. 一次任务完成后，前台不插入额外完成提示。
2. 用户通过分身最终回复自然感知任务结束。
3. 状态系统同步从执行态切换到完成态，并在短暂停留后回到空闲态。

### 14.2 错误反馈

1. 当任务失败、阻塞或无法继续时，由分身以自然语言回复用户。
2. 回复内容应优先说明当前失败原因、受阻环节或下一步建议。
3. 状态系统同步进入 `出错` 态。
4. 本期不在主聊天流中直接展示底层工具调用明细或技术堆栈。

## 15. 状态系统

状态系统用于在不同入口之间保持一致的任务感知，包括：

1. 菜单栏图标
2. 底部语音条
3. 主聊天页 Header

### 15.1 状态集合

本期统一采用以下 8 个状态：

1. `空闲`
2. `正在听`
3. `正在转写`
4. `等待发送`
5. `正在思考`
6. `正在执行`
7. `已完成`
8. `出错`

### 15.2 状态语义

1. `空闲`：分身当前未处理输入，也无进行中的任务。
2. `正在听`：语音条正在录音，分身正在接收用户语音输入。
3. `正在转写`：语音输入已结束，系统正在将语音转为文本。
4. `等待发送`：转写结果已生成，等待用户确认发送。该状态主要存在于语音条中。
5. `正在思考`：请求已提交，分身正在组织回答或准备执行动作。
6. `正在执行`：分身已进入实际任务执行阶段，但本期不向聊天流展示内部工具过程。
7. `已完成`：当前请求已形成最终回复。该状态短暂停留后回到 `空闲`。
8. `出错`：当前请求失败、受阻或无法继续。

### 15.3 状态展示原则

1. 菜单栏与主窗口使用同一套状态语义。
2. 允许展示形式不同，但不允许语义不一致。
3. 状态表达聚焦“用户可理解的任务阶段”。
4. 不在状态系统中暴露 memory 检索、prompt 注入、工具调用链等内部过程。

### 15.4 状态分布

1. `语音条` 重点承载：`正在听`、`正在转写`、`等待发送`
2. `主聊天页` 重点承载：`正在思考`、`正在执行`、`已完成`、`出错`
3. `菜单栏` 承载全局状态反馈，用于在主窗口未打开时向用户表达系统当前状态。

### 15.5 状态切换规则

建议本期采用以下主要迁移逻辑：

1. 语音输入路径：`空闲 -> 正在听 -> 正在转写 -> 等待发送 -> 正在思考/正在执行 -> 已完成 -> 空闲`
2. 主聊天输入路径：`空闲 -> 正在思考/正在执行 -> 已完成 -> 空闲`
3. 异常路径：任一处理中状态均可能进入 `出错`

## 16. 分身设定页

分身设定页用于定义 Shrew 作为单一 AI 分身的身份、人格与默认行为方式。该页面回答的是“Shrew 是谁”“Shrew 怎么说话”“Shrew 默认如何与我协作”，而不是系统配置问题。

本期分身设定采用中度设定模型，支持以下内容：

1. 名称
2. 头像
3. 一句简介
4. 性格风格
5. 说话语气
6. 默认回应方式
7. 工作偏好
8. 默认技能组合

系统提示词采用“半暴露”方案。默认情况下，用户通过结构化字段配置分身人格和行为偏好；原始 system prompt 作为高级设置存在，仅向有进阶控制需求的用户开放。该设计旨在优先传达“塑造一个分身”的产品心智，而不是让普通用户直接进行 prompt engineering。

### 16.1 字段结构

1. `名称`
2. `头像`
3. `一句简介`
4. `性格风格`
5. `说话语气`
6. `回答详略偏好`
7. `主动澄清偏好`
8. `工作方式偏好`
9. `默认技能`
10. `高级 system prompt`

### 16.2 交互原则

1. 默认优先展示结构化字段。
2. 高级 system prompt 不应作为主入口暴露。
3. 用户在该页的感受应是“我在塑造这个分身”，而不是“我在编辑一段 prompt”。

### 16.3 与其他页面的边界

1. 分身设定页不负责 API Key、模型连接、语音服务凭证等系统级配置。
2. 分身设定页不负责具体服务连接管理，该能力归属“服务连接”页。
3. 分身设定页不负责逐条管理 Memory，该能力归属 Memory 页。
4. 分身设定页只负责定义分身的身份、风格与默认协作习惯。

## 17. Memory 页

Memory 页用于帮助用户理解并修正 Shrew 的长期记忆。该页面不是聊天历史页，也不是技术型记忆数据库管理台，而是“分身当前如何理解我”的前台表达界面。

本页采用“关系档案型”设计，遵循：

1. 先帮助用户理解分身记住了什么
2. 再提供逐条修正能力

### 17.1 页面目标

1. 让用户理解 Shrew 当前记住了哪些长期信息
2. 让用户在必要时修正错误记忆
3. 让长期记忆服务持续关系，而不是制造不可控黑盒感

### 17.2 记忆模型

本期 Memory 采用混合型策略：

1. 系统自动记忆
2. 用户可查看与手动修正

系统不会实时把所有对话写入 Memory，而是在一次任务结束后，从本次任务相关内容中提炼候选长期记忆。

### 17.3 提炼时机

Memory 提炼发生在任务结束后。

系统可从以下信息中提炼记忆：

1. 用户长期偏好
2. 用户工作方式习惯
3. 项目长期背景
4. 稳定约束与规则
5. 用户明确纠正或强调的重要事实

### 17.4 不应进入 Memory 的内容

1. 一次性临时指令
2. 短期任务中间过程
3. 低价值闲聊内容
4. 工具调用过程细节
5. 明显只对当前单次任务有效的上下文

### 17.5 页面结构

Memory 页采用两层结构：

1. `记忆档案区`
2. `记忆卡片区`

### 17.6 记忆档案区

建议以主题化方式聚合展示，例如：

1. 用户偏好
2. 工作方式
3. 当前项目背景
4. 长期约束
5. 其他稳定事实

### 17.7 记忆卡片区

每条 memory 卡片建议至少包含：

1. 内容
2. 类型
3. 来源
4. 最近更新时间
5. 当前状态

### 17.8 用户可执行操作

本期建议支持以下操作：

1. 编辑
2. 删除
3. 标记失效
4. 置顶
5. 手动新增

### 17.9 与聊天流的关系

1. 聊天历史不等于 Memory。
2. Memory 是从任务完成后的交流中提炼出来的长期信息。
3. 用户不能通过修改聊天历史来修改 Memory。
4. 用户应通过 Memory 页直接修正长期记忆。

### 17.10 与 `/clear` 的关系

1. `/clear` 仅用于清空当前上下文。
2. `/clear` 不直接删除已有 Memory。
3. `/clear` 后系统从新的上下文继续理解用户，但长期 Memory 仍保持有效，除非用户在 Memory 页中主动修改。

## 18. 技能管理

技能管理页用于帮助用户理解“Shrew 当前启用了哪些技能”，并承接对底层 Claude Code skill 的真实配置流程。本页不应被设计为插件商店，也不应成为纯展示型页面，而应被定义为面向普通用户的技能配置入口：前台弱化技术术语，但保留真实可执行的配置流程。

### 18.1 页面目标

1. 让用户理解分身当前启用了哪些技能
2. 让用户能够查看、启用、停用和配置底层 Claude Code skill
3. 让技能配置服务于分身能力塑造，而不是形成抽象插件堆叠感

### 18.2 能力模型

本期技能采用混合型能力模型，由两部分组成：

1. 默认技能
2. 可启用附加技能

### 18.3 页面结构

1. `当前技能概览`
2. `已启用技能`
3. `可配置技能`
4. `技能配置流程`

### 18.4 参考交互模式

本页的交互模式可参考 Cursor 对 Rules 的管理方式：

1. 先展示“当前有哪些规则或技能正在生效”。
2. 再提供“新增或配置”的明确入口。
3. 对大多数用户先展示名称、说明、状态、作用范围等可理解信息。
4. 将原始配置内容、路径、底层格式等放入高级区域，而不是默认主视图。
5. 支持启用、停用、编辑和删除，但不把页面做成纯文本文件浏览器。

### 18.5 用户可执行操作

本期建议支持以下操作：

1. 查看技能说明
2. 启用技能
3. 停用技能
4. 配置技能所需参数
5. 调整默认优先使用范围

### 18.6 配置流程原则

技能管理页需要承接真实的 Claude Code skill 配置流程，但前台不应直接暴露过多底层技术数据。页面应做到：

1. 用户可以完成技能启用前所需的必要配置。
2. 当某个 skill 依赖额外参数、上下文或本地配置时，页面需要提供对应的配置入口。
3. 默认优先展示“这个技能能做什么”“当前是否可用”“还缺什么配置”。
4. 技术细节可折叠或收敛在高级信息中，而不是完全省略。

### 18.7 建议配置流程

建议采用类似 Cursor Rules 的“列表 + 编辑流”结构：

1. 用户进入技能管理页后，先看到当前已启用技能列表。
2. 每个技能项展示：名称、说明、状态、来源、配置是否完整。
3. 用户可点击 `新增技能` 进入选择流程。
4. 选择一个技能后，进入配置页或抽屉，填写必要参数。
5. 配置完成后，用户可保存并启用该技能。
6. 已启用技能支持后续再次编辑、停用或删除。
7. 高级配置可通过“查看底层配置”或“高级设置”折叠展开。

### 18.8 建议页面元素

建议页面包含以下元素：

1. `新增技能` 主按钮
2. `已启用`、`未启用`、`配置异常` 等状态筛选
3. 技能列表项
4. 技能详情面板或右侧抽屉
5. 高级配置折叠区

### 18.9 页面表达原则

1. 技能管理页优先使用“Shrew 当前擅长”“Shrew 已启用的技能”这类分身导向表达。
2. 不以插件商店式设计为主。
3. 不以前台暴露复杂技术术语为目标，但必须保留真实配置链路。

## 19. 服务连接

服务连接页用于管理 Shrew 接入的外部服务来源，并承接对底层 Claude Code MCP 的真实配置流程。该页面回答的是：

`Shrew 连接了哪些外部服务`

而不是：

`Shrew 具备哪些技能`

### 19.1 页面目标

1. 让用户理解分身当前连接了哪些外部服务
2. 让用户完成底层 Claude Code MCP 的配置、连接和授权流程
3. 让外部服务能力的可用性变得可见、可控

### 19.2 页面结构

1. `已连接服务`
2. `连接状态`
3. `服务配置流程`
4. `连接管理动作`

### 19.3 参考交互模式

本页的交互模式可参考 Cursor 对 MCP 的管理方式：

1. 先展示当前已配置的 MCP 服务列表。
2. 每个服务项展示连接状态与可用性。
3. 提供 `新增服务连接` 的明确入口。
4. 新增时通过配置向导或表单完成必要参数填写。
5. 默认视图弱化底层协议细节，但保留高级配置查看与编辑入口。

### 19.4 用户可执行操作

本期建议支持：

1. 新增连接
2. 填写或更新连接配置
3. 断开连接
4. 重新连接
5. 重新授权
6. 查看该服务提供的帮助说明

### 19.5 配置流程原则

服务连接页需要承接真实的 Claude Code MCP 配置流程，但前台不应直接暴露协议细节堆栈。页面应做到：

1. 用户可以完成服务连接所需的必要配置与授权。
2. 当某个 MCP 依赖地址、命令、凭证或其他连接参数时，页面需要提供对应的配置入口。
3. 默认优先展示“这个服务能帮 Shrew 做什么”“当前是否已连接”“还缺什么配置”。
4. 技术参数可以折叠在高级信息中，而不是完全隐藏。

### 19.6 建议配置流程

建议采用类似 Cursor MCP Settings 的“列表 + 新增向导 + 高级配置”结构：

1. 用户进入服务连接页后，先看到当前服务列表。
2. 每个服务项展示：服务名称、说明、状态、最近一次检查结果。
3. 用户点击 `新增服务连接` 后，选择服务类型或自定义连接方式。
4. 系统引导用户填写必要配置，例如地址、命令、凭证、授权信息等。
5. 完成配置后执行一次连接校验。
6. 校验成功后，服务进入 `已连接` 状态。
7. 若校验失败，页面需明确指出缺失配置或错误原因，并允许继续编辑。
8. 高级配置可通过折叠区查看底层配置映射，但默认不强展示。

### 19.7 建议页面元素

建议页面包含以下元素：

1. `新增服务连接` 主按钮
2. `已连接`、`未连接`、`异常`、`需重新授权` 状态筛选
3. 服务列表项
4. 服务详情面板或右侧抽屉
5. 连接测试按钮
6. 高级配置折叠区

### 19.8 页面表达原则

1. 页面应优先表达“Shrew 接入了哪些外部世界能力来源”。
2. 不在默认视图中强调协议、schema、transport 等技术细节。
3. 默认文案以结果导向表达。
4. 可弱化技术细节展示，但不可省略真实配置流程。

### 19.9 与技能管理页的边界

1. 技能管理页回答“Shrew 会什么”。
2. 服务连接页回答“Shrew 连到了什么”。
3. 本期不引入 GitHub、Notion 等“独立能力层”概念，避免中间抽象混淆。

## 20. 设置页

设置页用于管理系统级配置，回答：

`这个应用如何运行`

而不是：

`这个分身是什么样`

### 20.1 页面目标

1. 管理应用运行所需的系统配置
2. 降低凭证、模型、语音和权限配置的复杂度
3. 让用户可以快速定位高频系统设置项

### 20.2 页面结构

设置页采用单页分组卡片式结构，不再引入设置内部二级导航。

建议包含以下卡片：

1. `模型与连接`
2. `语音`
3. `快捷键与唤起`
4. `工作目录与执行权限`
5. `数据与隐私`

### 20.3 页面原则

1. 设置页只承载系统级配置。
2. 页面按用户任务组织，而不是按底层技术对象拆分。
3. 高频项优先，低频高级项后置。
4. 默认保持克制，不引入过深层级。

## 21. 功能需求

### 21.1 主窗口

1. 系统必须支持通过右上角应用入口打开主窗口。
2. 主窗口首层必须直接进入聊天页。
3. 主窗口不得默认展示会话列表作为首屏主体。
4. 主窗口必须支持进入以下二级页：`分身设定`、`Memory`、`技能管理`、`服务连接`、`设置`

### 21.2 聊天页

1. 聊天页必须包含顶部 Header、聊天流、底部输入框三部分。
2. Header 必须展示头像、名称、状态点和一行副信息。
3. Header 副信息必须支持空闲态显示人格描述、忙碌态显示状态描述。
4. Header 必须支持点击展开身份卡。
5. 聊天流必须采用连续时间流展示。
6. 聊天流本期只允许展示：`用户消息`、`分身消息`、`必要系统消息`
7. 聊天流不得展示工具调用过程或技术执行明细。
8. 输入框必须常驻。
9. 输入框必须支持自然语言输入。
10. 输入框必须支持 slash 命令机制。
11. 本期 slash 命令必须至少支持 `/clear`。
12. 输入框必须支持 `Enter` 发送、`Shift+Enter` 换行。
13. 用户发送后，用户消息必须立即进入聊天流。
14. 分身回复必须支持流式生成。
15. 空状态必须展示欢迎语与简短自我介绍。
16. 空状态不得展示推荐问题或模板起手式。

### 21.3 `/clear`

1. 系统必须支持在输入框中识别 `/clear` 命令。
2. 执行 `/clear` 后，系统必须清空当前上下文。
3. 执行 `/clear` 后，系统不得创建一个显式新会话供用户切换。
4. 执行 `/clear` 后，聊天流必须插入一条系统消息。
5. `/clear` 不得直接删除已有 Memory。

### 21.4 语音条

1. 系统必须支持通过全局快捷键唤起底部悬浮语音条。
2. 语音条必须支持录音、转写、编辑、回车确认发送。
3. 转写完成后，语音条必须进入 `等待发送` 状态。
4. 用户确认发送前，不得向主聊天流写入用户消息。
5. 用户确认发送后，语音内容必须以用户消息形式写入主聊天流。
6. 语音条发送后，系统不得自动打开主窗口。

### 21.5 状态系统

1. 系统必须支持以下 8 个状态：`空闲`、`正在听`、`正在转写`、`等待发送`、`正在思考`、`正在执行`、`已完成`、`出错`
2. 菜单栏与主窗口必须使用同一套状态语义。
3. 菜单栏必须在主窗口未打开时提供状态反馈。
4. 任务完成后，状态必须短暂停留在 `已完成`，随后回到 `空闲`。
5. 出错时，状态必须进入 `出错`。

### 21.6 分身设定

1. 系统必须支持以下结构化字段：`名称`、`头像`、`一句简介`、`性格风格`、`说话语气`、`回答详略偏好`、`主动澄清偏好`、`工作方式偏好`、`默认技能`
2. 系统必须提供高级 system prompt 设置入口。
3. 高级 system prompt 必须默认折叠，不得作为主入口。
4. 分身设定页不得承载 API Key、模型连接等系统级配置。

### 21.7 Memory

1. 系统必须在任务结束后进行 Memory 提炼。
2. Memory 不得对所有中间过程进行实时写入。
3. Memory 页必须包含汇总档案视图。
4. Memory 页必须包含逐条卡片管理视图。
5. 用户必须能够对 Memory 执行：`编辑`、`删除`、`标记失效`、`置顶`、`手动新增`
6. Memory 页必须能展示每条 Memory 的内容、类型、来源、更新时间和状态。

### 21.8 技能管理

1. 技能管理页必须展示当前技能概览。
2. 技能管理页必须展示已启用技能。
3. 技能管理页必须支持技能启用与停用。
4. 技能管理页必须支持底层 Claude Code skill 的必要配置流程。
5. 技能管理页必须区分默认技能与后续启用技能。
6. 技能管理页不得被设计为插件商店主视角。

### 21.9 服务连接

1. 服务连接页必须展示已连接服务。
2. 每个服务项必须展示名称、状态和帮助说明。
3. 服务连接页必须支持底层 Claude Code MCP 的必要配置流程。
4. 服务连接页必须支持：`新增连接`、`断开连接`、`重新连接`、`重新授权`
5. 服务连接页默认不得暴露底层协议细节，但不得省略用户完成配置所需的信息与步骤。

### 21.10 设置

1. 设置页必须采用单页分组卡片式结构。
2. 设置页必须至少包含以下卡片：`模型与连接`、`语音`、`快捷键与唤起`、`工作目录与执行权限`、`数据与隐私`
3. 设置页不得承载分身人格与 Memory 管理。

## 22. 非功能需求

### 22.1 性能

1. 主窗口打开应尽量快速，避免明显白屏或长时间等待。
2. 聊天页滚动和流式回复应保持顺滑。
3. 语音条唤起应足够轻量，不应有明显卡顿。

### 22.2 稳定性

1. 主窗口、语音条、菜单栏状态之间的状态同步必须稳定。
2. 请求执行失败时，系统必须能进入可恢复状态，而不是卡死在处理中。
3. 长时间运行后，历史消息和状态展示不能出现明显错乱。

### 22.3 数据安全

1. API Key、Token、语音服务凭证必须安全存储。
2. 本地数据存储位置必须清晰可控。
3. 用户修改或删除 Memory 后，后续使用必须反映该结果。

### 22.4 隐私

1. 用户应能够理解哪些信息会进入长期 Memory。
2. 用户应能够清理本地数据。
3. 系统不应在前台误导用户，以为 `/clear` 会清除长期 Memory。

### 22.5 可理解性

1. 页面命名必须以用户可理解语言为主。
2. 前台应优先使用“技能”“服务连接”等表达，而不是底层技术术语。
3. 错误反馈必须尽量可行动，而不是只输出技术异常。

## 23. 验收标准

### 23.1 主体验验收

1. 用户点击右上角入口后，能够直接进入聊天页。
2. 聊天页包含 Header、聊天流、常驻输入框。
3. 用户能通过文字持续聊天。
4. 用户能通过语音条快捷发起请求。
5. 语音条发送后不会自动打开主窗口。

### 23.2 上下文与命令验收

1. 输入 `/clear` 后，当前上下文被清空。
2. 聊天流中出现一条系统消息提示 `/clear` 已生效。
3. `/clear` 后用户仍处在同一条持续聊天流中。
4. `/clear` 不会让既有 Memory 消失。

### 23.3 Memory 验收

1. 一次任务结束后，系统可产出长期记忆结果。
2. 用户可查看汇总档案视图。
3. 用户可逐条编辑、删除、标记失效、置顶或新增 Memory。
4. 用户修改后的 Memory 会影响后续长期记忆使用。

### 23.4 状态验收

1. 语音输入时可观察到 `正在听 -> 正在转写 -> 等待发送`。
2. 请求提交后可观察到 `正在思考/正在执行`。
3. 请求完成后可观察到 `已完成 -> 空闲`。
4. 出错时可观察到 `出错` 状态，并收到分身自然语言说明。

### 23.5 命名与信息架构验收

1. 用户能区分“分身设定”“技能管理”“服务连接”“设置”的职责。
2. 用户不会在首屏被引导去管理会话列表。
3. 用户在空状态下感知到的是“一个正在等待交流的分身”，而不是功能面板。

## 25. 字段级配置定义

### 25.1 分身设定字段定义

1. `名称`
   - 控件类型：单行输入框
   - 默认值：`Shrew`
   - 必填：是
   - 规则：不能为空，长度建议 1 到 30 字

2. `头像`
   - 控件类型：头像选择器或上传器
   - 默认值：系统默认头像
   - 必填：否

3. `一句简介`
   - 控件类型：单行或短多行输入框
   - 默认值：空
   - 规则：长度建议不超过 80 字

4. `性格风格`
   - 控件类型：单选或枚举选择
   - 选项建议：`冷静`、`温和`、`直接`、`专业`、`鼓励式`
   - 默认值：`专业`

5. `说话语气`
   - 控件类型：单选或枚举选择
   - 选项建议：`简洁`、`自然`、`克制`、`亲和`
   - 默认值：`自然`

   - 控件类型：单选

   - 默认值：`平衡`

## 26. 本地持久化策略

Shrew 本期定位为对 Claude Code 的桌面端产品化封装，因此在持久化策略上遵循“最小壳层持久化、优先复用 Claude Code 既有能力”的原则。本期不预设数据库作为必选实现，不以自建完整会话存储系统为目标。

### 26.1 总体原则

1. 优先复用 Claude Code 已有的会话、上下文和执行能力。
2. Shrew 壳层只持久化 Claude Code 不直接负责、但产品层必须存在的信息。
3. 本期不要求引入数据库。
4. 本期优先采用本地结构化文件与安全存储完成最小可用持久化。

### 26.2 需要持久化的信息

本期建议持久化以下几类本地数据：

1. 分身设定
2. 系统设置
3. 长期 Memory
4. 服务连接状态
5. 技能启用状态

### 26.3 优先复用 Claude Code 的信息

以下内容本期建议优先复用 Claude Code 自身能力，不在 Shrew 壳层重复构建完整存储系统：

1. 聊天上下文管理
2. 任务执行过程
3. Agent 调用链
4. 会话恢复基础能力
5. 工具调用与底层执行逻辑

### 26.4 本期不建议的实现方式

本期不建议将以下能力作为首要目标自建：

1. 完整的数据库式聊天历史系统
2. 完整的任务执行日志数据库
3. 与 Claude Code 平行的第二套会话管理体系
4. 仅用单一 Markdown 文件作为运行时持久化数据源

### 26.5 推荐的最小持久化方案

1. `persona.json`
2. `settings.json`
3. `memory.json`
4. `service-connections.json`
5. `skills.json`
6. `secure storage`

### 26.6 持久化边界

1. 聊天流的产品展示可以由 Shrew 维护自己的前台映射结构，但不要求自建完整消息数据库。
2. 长期 Memory 属于 Shrew 产品能力，应由 Shrew 自主持久化。
3. 分身设定与系统设置属于 Shrew 产品配置，应由 Shrew 自主持久化。
4. 敏感信息必须与普通配置分离存储，不进入普通本地明文配置文件。

## 28. 状态流转与对象关系

### 28.1 顶层对象关系

1. `Persona` 拥有 `ChatMessage`、`Task`、`MemoryItem`、`SkillState`、`ServiceConnectionState`
2. `Task` 由一条用户消息触发，最终关联一条分身最终回复
3. `MemoryItem` 通常来源于一个已结束的 `Task`

### 28.2 前台聊天流与后台任务关系

1. 一条用户消息触发一个任务。
2. 一个任务至少有一条用户消息。
3. 一个任务通常有一条最终分身回复。
4. 一个任务结束后，可产出 0 到多条 Memory。

### 28.3 语音路径状态流

1. `空闲`
2. 用户按快捷键后进入 `正在听`
3. 录音结束进入 `正在转写`
4. 转写完成进入 `等待发送`
5. 用户回车确认后进入 `正在思考` 或 `正在执行`
6. 最终进入 `已完成 -> 空闲` 或 `出错`

### 28.4 文字路径状态流

1. `空闲`
2. 用户发送文字后立即写入用户消息
3. 进入 `正在思考` 或 `正在执行`
4. 分身回复流式生成
5. 最终进入 `已完成 -> 空闲` 或 `出错`
